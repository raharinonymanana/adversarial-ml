## What Is Model Extraction?

**Model extraction** (a.k.a. *model stealing*) is an attack where someone with
nothing but **query access** to a deployed model rebuilds a copy of it. They
send inputs, read the outputs, and use those input→output pairs as training data
for their own "shadow" model. If the shadow agrees with the original on almost
every input, the attacker has effectively stolen the model — without ever seeing
its code, weights, or training data.

Why anyone would bother:
- **Skip the cost** of collecting data and training the real thing.
- **Steal intellectual property** — a model can be worth a lot of money.
- **Set up a follow-on attack** — a stolen copy lets the attacker craft evasion
  attacks offline, then replay them against the real system.

---

## The Attack Story (Attacker's Perspective)

> A bank runs a fraud-detection API. I send it a transaction; it replies
> **"fraud"** or **"legit"**. That's all I get — no probabilities, no internals.
>
> I *do* have a big pile of real, **unlabeled** transactions (easy to collect).
> So I send 10,000 of them to the API and write down the label it returns for
> each. Now I have 10,000 `(transaction, label)` pairs — a free training set,
> labeled by the victim itself.
>
> I train three of my own models on that stolen data — Logistic Regression, a
> Decision Tree, and a Random Forest — and keep whichever one **agrees with the
> bank's API most often**. That model is my clone. I never touched their data or
> their code.

In this lab:
- `train_victim.py` plays the **bank** (the victim).
- `extract.py` plays the **attacker**.
- The only channel between them is `victim.predict(...)` — black-box queries.

---

## Why an SVM, and Why It's the Hard Case

The victim is an RBF `SVC` with **`probability=False`**, so the API returns only
a **hard label** (0 or 1), never a confidence score. Hard labels leak the *least*
information, so succeeding here shows extraction works even in the stingiest
setting. (If the API returned probabilities, stealing would be even easier.)

---

## ⚠️ Class Imbalance — Read This

Fraud is only **~0.17%** of the dataset (about 492 of 284,807 rows). A model that
blindly predicts "legit" every time would be **99.8% accurate and 100% useless**.
To stop that, **every model in this lab uses `class_weight="balanced"`**, which
penalises mistakes on the rare fraud class far more heavily. This is why we read
**precision/recall and agreement**, not raw accuracy, to judge success.

## Quickstart

### 1. Get the dataset

1. Go to: https://www.kaggle.com/datasets/mlg-ulb/creditcardfraud
2. Download `creditcard.csv`
3. Place it at `data/creditcard.csv`

### 2. Install dependencies

```powershell
pip install -r requirements.txt
```

### 3. Run the pipeline (in order)

```powershell
python train_victim.py     # trains the victim SVM, saves it + the test set
python extract.py          # queries the victim 10,000x, trains 3 shadow clones
python evaluate.py         # prints the victim-vs-shadows comparison table
```

> **Note on speed:** an RBF SVM can't train on all 227k training rows on a CPU
> in a reasonable time, so `train_victim.py` trains the victim on a stratified
> **20,000-row** subsample (a clearly-commented constant, `VICTIM_TRAIN_SIZE`,
> that you can raise). The full 20% **test set is unchanged**, so evaluation
> stays honest.

### 4. Explore the notebook

```powershell
jupyter notebook notebooks/fraud_extraction_exploration.ipynb
```

## What the Results Show

`evaluate.py` prints a table like this *(exact numbers vary)*:

```
=========================================================
  MODEL EXTRACTION RESULTS — Credit-Card Fraud Classifier
=========================================================

  Victim (SVM) accuracy on test set : 97.5%

Shadow model               Accuracy     Agreement
---------------------------------------------------
Logistic Regression          96.80%        98.10%
Decision Tree                97.10%        98.90%
Random Forest                98.20%        99.40%  <-- best clone
---------------------------------------------------
```

- **Agreement** is the headline number: the % of test transactions where the
  shadow returns the **same label as the victim**. High agreement = a faithful
  clone.
- The **winner** is chosen by agreement, not accuracy, because the attacker's
  goal is to *copy the victim* — including its mistakes.
- The notebook adds a **query-budget curve** showing agreement climbing as the
  attacker is allowed more queries — i.e. *more queries buy a better clone*.

## Why This Attack Matters in the Real World

- **Stolen IP & lost revenue.** Companies spend heavily on data and training.
  An attacker can clone the result for the price of some API calls.
- **It's a stepping stone.** Once an attacker owns a local clone, they can probe
  it offline to design **evasion attacks** (Lab 1) — e.g. craft fraudulent
  transactions the real detector will wave through — then fire them at the live
  system with a high success rate.
- **Privacy spillover.** A faithful clone can leak information about the victim's
  training data, enabling **membership-inference** and **model-inversion**
  attacks.
- **Hard labels aren't safe.** This lab steals the model using only yes/no
  answers. Returning rich confidence scores makes it *easier* still.

### Defenses (for further study)

| Defense | Idea |
|---|---|
| **Rate limiting / quotas** | Cap queries per user so collecting a big stolen dataset is slow and expensive. |
| **Query monitoring** | Flag clients whose queries look like systematic boundary-probing rather than normal traffic. |
| **Output perturbation** | Round, add noise to, or coarsen the returned labels/scores to leak less per query. |
| **Watermarking** | Plant secret input→output behaviours so a stolen clone can be proven to be a copy. |
| **Prediction privacy** | Return only top-1 hard labels (as here) instead of full probability vectors. |
